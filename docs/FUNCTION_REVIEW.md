# 기능 분류와 보완점

DevPilot의 중심 기능은 `프로젝트 -> 일감 -> workflow`이다. 아래 분류는 실사용 빈도와 핵심 흐름 기여도를 기준으로 정리했다.

## 핵심 기능

| 기능 | 역할 |
| --- | --- |
| 프로젝트 관리 | 수동 관리 프로젝트와 Jira 연결 프로젝트를 만들고 업무 단위를 분리한다. |
| 프로젝트별 일감 부여 | 프로젝트 카드에서 직접 일감을 등록하거나 Jira 일감을 가져온다. |
| 일감 처리 workflow | AI 분석, repository 확정, workspace 생성, 구현, 테스트, 보고를 단계별 승인 흐름으로 관리한다. |
| AI 작업 지휘관 | 일감 상태를 바탕으로 분석, 작업 계획, repo 후보, 브랜치 전략, 테스트 추천, 보고 초안을 정리한다. |
| Git/workspace 관리 | 일감 단위 worktree, 브랜치, 변경 파일, 테스트/보고 근거를 연결한다. |
| 기록/보고 | 테스트 결과, 작업 보고, 메모를 날짜별 기록과 브리핑 근거로 남긴다. |

## 선택 기능

| 기능 | 분류 이유 |
| --- | --- |
| Jira 연동 | Jira를 쓰면 강력하지만, 수동 일감 등록만으로도 workflow는 동작해야 한다. |
| Slack 연동 | 보고/브리핑 공유용이며 일감 처리 자체의 필수 조건은 아니다. |
| Codex 프로젝트/스레드 전체 조회 | 일감별 Codex 연결은 중요하지만 전체 목록 조회는 보조 도구에 가깝다. |
| 토큰/연동 상태 | 문제 진단용 화면이며 상시 작업 화면보다는 설정/연동 상태 영역에 두는 것이 적합하다. |
| 연장근무 기록 | 개인 기록으로 유용하지만 핵심 개발 workflow와는 분리된 부가 기능이다. |

## 사용 빈도가 낮아 보이는 기능

| 기능 | 정리 방향 |
| --- | --- |
| 팀 Jira 흐름 전체 조회 | 개인 일감 처리 화면보다 우선순위가 낮으므로 보조 메뉴로 유지한다. |
| 브리핑 단독 화면 | 별도 화면보다 workflow와 기록에서 자동 생성되는 결과물로 두는 편이 자연스럽다. |
| Codex 스레드 전체 탐색 | 선택한 일감의 스레드만 보여주는 방식이 실사용에 더 가깝다. |
| 토큰 만료 상세 | 문제 발생 시 확인하는 진단 기능으로 축소한다. |
| 연장근무 상세 설정 | 모달 또는 기록 하위 기능으로 유지한다. |

## 보완 필요 기능

| 우선순위 | 보완점 |
| --- | --- |
| 높음 | 프로젝트 카드에 승인 대기, 테스트 필요, 보고 필요 상태를 더 명확히 표시한다. |
| 높음 | AI 작업 지휘관 결과를 repository 선택, 브랜치 전략, 테스트 명령 추천 버튼과 직접 연결한다. |
| 높음 | Jira 비활성 환경에서는 Jira 문구와 경고를 줄이고 수동 workflow 중심 문구를 강화한다. |
| 중간 | workflow 단계별 승인/완료 버튼의 상태 전환 규칙과 문구를 통일한다. |
| 중간 | rebase 충돌, dirty worktree, workspace cleanup 실패 상황의 복구 가이드를 제공한다. |
| 중간 | 테스트 결과, 커밋, 브랜치, AI 분석 결과가 보고 초안으로 자동 모이는 흐름을 강화한다. |
| 낮음 | SwiftUI 대형 화면을 기능 단위 컴포넌트로 더 분리하고 화면 단위 테스트를 보강한다. |
| 낮음 | 주간/월간 회고 기능을 기록 화면의 확장 기능으로 검토한다. |
